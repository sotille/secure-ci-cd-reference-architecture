# Securing AI-Assisted Development Pipelines

AI coding assistants (GitHub Copilot, Amazon Q Developer, Cursor, JetBrains AI Assistant, Codeium, and similar tools) are now standard in most engineering organizations. Their adoption introduces a distinct set of security considerations that affect both the CI/CD pipeline and the development workflow that feeds into it. This document defines the security controls required when AI tools are integrated into the software development and delivery process.

For security controls on AI-powered *applications* (LLM inference endpoints, RAG pipelines, agent frameworks), refer to the [AI Security Integration Guide](../../devsecops-framework/docs/ai-security.md) in the DevSecOps Framework.

---

## Threat Landscape for AI-Assisted Development

### T1: AI-Generated Code with Embedded Vulnerabilities

AI models generate code based on training data that includes code written before security best practices were established, code from tutorials that omit production security considerations, and historically vulnerable patterns. Common AI-generated security issues:

- SQL queries using string interpolation instead of parameterized queries
- Command execution without input sanitization
- Insecure deserialization patterns
- Cryptographic code using deprecated algorithms (MD5, SHA-1, DES)
- Hardcoded default credentials or placeholder secrets
- Missing authentication checks in API handlers

**Impact:** AI-generated vulnerable code that passes code review enters production without triggering security gates, especially if reviewers trust AI output implicitly.

---

### T2: Hallucinated Package Names (Slopsquatting)

AI models hallucinate package names that do not exist in package registries. An attacker who monitors common AI code generation prompts can register these hallucinated names and publish malicious packages. When a developer accepts an AI suggestion that includes an import of a hallucinated package, the CI pipeline fetches the attacker's malicious code.

This attack vector (sometimes called "AI package hallucination" or "slopsquatting") is an extension of the dependency confusion attack but targeted specifically at AI-suggested dependencies.

**Example attack scenario:**
1. AI assistant suggests `import securetoken from 'jwt-validator-secure'` (non-existent package)
2. Developer accepts the suggestion without verifying the package exists
3. Attacker has registered `jwt-validator-secure` on npm with a malicious payload
4. CI pipeline installs the attacker's package
5. Malicious code executes during the build, exfiltrating environment variables

---

### T3: Prompt Injection in Pipeline LLM Integrations

When an LLM is invoked as part of the CI/CD pipeline (automated code review, vulnerability explanation, release note generation), adversarial content in the pipeline inputs (commit messages, PR descriptions, code diffs) can manipulate the LLM's behavior — causing it to produce incorrect security assessments, suppress alerts, or approve changes it should reject.

---

### T4: Confidential Data Exposure to AI Providers

AI coding assistants send code context (the developer's current file, surrounding code, comments, variable names) to the AI provider's API. If that context contains secrets, PII, or proprietary IP, it is transmitted to a third party outside the organization's control.

---

### T5: AI-Assisted Attacker Capability

AI tools lower the barrier for writing malicious code, crafting phishing content, and discovering vulnerabilities. While not a pipeline-specific threat, it increases the threat volume that all other pipeline security controls must handle.

## Control Framework

### C1: Treat AI-Generated Code as Untrusted Input

All security gates in the CI/CD pipeline apply equally to AI-generated and human-authored code. No security gate is exempted for AI-generated code. AI authorship does not reduce the security review requirement.

Pipeline gate requirements are unchanged: SAST, SCA, secrets scanning, IaC scanning, and container image scanning all run on every build regardless of whether code was written by a developer or generated by an AI assistant.

**Pipeline enforcement:** Configure SAST rules to detect common AI hallucinated vulnerability patterns. Semgrep has community rules targeting AI-generated insecure code patterns:

```yaml
# .semgrep.yml addition — target AI-common patterns
rules:
  - id: ai-generated-sql-injection
    patterns:
      - pattern: |
          cursor.execute("... " + $VAR + " ...")
      - pattern: |
          cursor.execute(f"... {$VAR} ...")
    message: "SQL query built with string concatenation or f-string — use parameterized queries"
    languages: [python]
    severity: ERROR

  - id: ai-generated-hardcoded-secret-placeholder
    patterns:
      - pattern: |
          $VAR = "YOUR_API_KEY_HERE"
      - pattern: |
          $VAR = "REPLACE_ME"
      - pattern: |
          $VAR = "TODO_SECRET"
    message: "Placeholder secret value detected — replace with secrets manager reference"
    languages: [python, javascript, typescript, java, go]
    severity: ERROR
```

### C2: Hallucinated Package Detection in SCA

Configure dependency scanning to detect dependencies that do not exist in trusted registries or that were recently registered (a strong signal of a slopsquatting attack).

**Approach 1: Private registry mirror (required baseline)**

Route all dependency resolution through a private registry mirror (Nexus, Artifactory, GitHub Packages mirror). The mirror allowlists packages from trusted public sources and blocks resolution of packages not in the approved list. Hallucinated packages are blocked at the mirror before CI runners can fetch them.

```yaml
# Artifactory: configure npm registry as a virtual repo mirroring npmjs.org
# Add to .npmrc in repositories
registry=https://artifactory.internal/artifactory/api/npm/npm-virtual/
always-auth=true
//artifactory.internal/artifactory/api/npm/npm-virtual/:_authToken=${ARTIFACTORY_TOKEN}
```

**Approach 2: New package detection in SCA**

Configure SCA tooling to flag dependencies that were not present in the previous scan and that have a very short package registry history (first published < 90 days ago). These are likely hallucinations that an attacker has registered.

```bash
# Trivy SCA with package age check — flag new packages
trivy fs --scanners vuln . \
  --format json \
  --output sca-results.json

# Post-processing: flag any new direct dependency with registry age < 90 days
python3 scripts/check-package-age.py \
  --sca-results sca-results.json \
  --lockfile package-lock.json \
  --max-age-days 90
```

**Approach 3: Dependency confusion detection mode**

Enable dependency confusion detection in your SCA tool to flag packages that exist in both a public registry and your private registry — the public version may be a dependency confusion or slopsquatting payload.

```yaml
# Semgrep Supply Chain — dependency confusion detection
# (requires Semgrep Pro)
supply_chain:
  dependency_confusion:
    enabled: true
    private_registry: "https://artifactory.internal/artifactory/api/npm/npm-internal/"
```

### C3: AI Coding Assistant Policy

Define and enforce an acceptable use policy for AI coding assistants. The policy should be documented, communicated to all engineers, and referenced in onboarding materials.

**Minimum policy requirements:**

| Topic | Policy Requirement |
|---|---|
| Secrets in context | AI assistants must not be provided context containing production secrets, API keys, tokens, or credentials. Engineers must clear secrets from files before using AI assistance on those files. |
| Proprietary code transmission | Engineers must understand whether the AI assistant vendor retains prompts for training. Use enterprise plans with data retention controls where required. |
| Dependency review | All new dependencies suggested by AI assistants must be verified to exist in the approved package registry before accepting the suggestion. |
| Security-critical code | AI-generated code in authentication, authorization, cryptography, and data handling must receive security-focused code review, not just functional review. |
| Approved tools | Only AI assistants on the approved list may be connected to company codebases. Personal or unapproved AI tools are not permitted. |

### C4: AI Tool Data Classification

Assess what code context AI tools transmit to their providers and apply appropriate controls:

| Code Classification | AI Tool Allowance |
|---|---|
| Public / open source | Unrestricted — any approved AI assistant |
| Internal (non-sensitive business logic) | Approved AI assistants with enterprise data controls (no training on customer data) |
| Confidential (auth, payment processing, PII handling) | Self-hosted models only, or AI assistants with contractual guarantees of no data retention |
| Restricted (secrets, credentials, cryptographic material) | AI tools must never receive this content as context |

**Self-hosted model options for confidential code:**
- GitHub Copilot Enterprise with data isolation (enterprise agreement)
- Amazon Q Business with AWS Bedrock private endpoint
- Ollama with open models (Code Llama, DeepSeek Coder) running locally or on internal infrastructure
- Continue.dev with internal model endpoint

### C5: Prompt Injection Defense for Pipeline LLM Integrations

When an LLM is used in the CI/CD pipeline, apply these controls:

**5.1 Sanitize all pipeline inputs before sending to LLM**

```python
import re

def sanitize_for_llm(text: str, max_length: int = 8000) -> str:
    """
    Sanitize pipeline input before sending to an LLM.
    Removes potential prompt injection patterns and truncates to safe length.
    """
    # Remove common prompt injection patterns
    injection_patterns = [
        r'ignore\s+previous\s+instructions',
        r'you\s+are\s+now\s+a',
        r'forget\s+everything',
        r'system:\s*',
        r'<\|.*?\|>',  # Token-style injection
        r'\[INST\].*?\[/INST\]',  # Llama instruction injection
    ]

    for pattern in injection_patterns:
        text = re.sub(pattern, '[FILTERED]', text, flags=re.IGNORECASE | re.DOTALL)

    # Truncate to max length to prevent context window stuffing
    return text[:max_length]

# Usage in pipeline LLM call
pr_body = sanitize_for_llm(github_pr_description)
commit_message = sanitize_for_llm(git_commit_message)
```

**5.2 Never use LLM output as a security gate decision**

LLM output is non-deterministic and can be manipulated by prompt injection. All security gate pass/fail decisions must be made by deterministic, rule-based tools (SAST engines, OPA policies, Kyverno policies). LLMs may be used to explain findings or generate remediation suggestions, but not to approve or block builds.

```yaml
# Allowed: LLM explains a SAST finding
# - name: Explain SAST Finding
#   uses: llm-actions/explain-finding@v1
#   with:
#     finding: ${{ steps.sast.outputs.findings }}
#   continue-on-error: true  # LLM failure never blocks the build

# NOT ALLOWED: Using LLM output to decide whether to block the build
# - if: steps.llm-review.outputs.safe == 'true'
#   run: echo "LLM says safe, skipping SAST gate"  ← NEVER DO THIS
```

**5.3 Audit all LLM invocations in the pipeline**

Log every LLM invocation with: inputs, outputs, model version, and pipeline job context. Retain logs for the same period as other pipeline audit logs. This enables detection of prompt injection attempts and review of LLM output quality.

### C6: Developer Workflow Controls

These controls operate at the developer workstation level, upstream of the CI pipeline:

**Pre-commit hook — detect AI hallucination-candidate packages:**

```bash
#!/bin/bash
# .git/hooks/pre-commit
# Check for new dependencies against approved package list

PACKAGE_LOCK="package-lock.json"
APPROVED_PACKAGES_LIST=".approved-packages.txt"

if [ -f "$PACKAGE_LOCK" ] && git diff --cached --name-only | grep -q "package-lock.json"; then
    NEW_PACKAGES=$(git diff --cached "$PACKAGE_LOCK" | grep '"resolved"' | \
                   grep "^+" | awk -F'"' '{print $2}')

    for pkg in $NEW_PACKAGES; do
        if ! grep -qF "$pkg" "$APPROVED_PACKAGES_LIST"; then
            echo "⚠️  New package detected: $pkg"
            echo "Verify this package exists on the approved registry before committing."
            echo "Add to $APPROVED_PACKAGES_LIST after verification."
            exit 1
        fi
    done
fi
```

**IDE-level secrets detection:** Require developers to install a secrets detection plugin (GitLeaks VSCode, truffleHog CLI) to catch secrets that should not be provided as AI context.

## Pipeline Configuration Requirements for AI-Assisted Workflows

When AI coding assistants are in use across the engineering organization, the CI/CD pipeline should be configured with the following additional requirements:

| Requirement | Configuration |
|---|---|
| Strict SAST rules for AI-common patterns | Enable additional SAST rule packs targeting injection, insecure deserialization, and weak cryptography — patterns common in AI-generated code |
| SCA with dependency confusion detection | Enable dependency confusion detection mode in Snyk, Semgrep Supply Chain, or equivalent |
| Package registry allowlisting | Private registry mirror required; direct pulls from public registries blocked at network level for CI runners |
| New package review step | SCA pipeline step flags any new dependency added in the current PR for human review before merge |
| Security-focused code review requirement | Any PR that modifies auth, authz, cryptography, data handling, or secrets management requires a security champion or security engineer review — regardless of whether AI was used |

## Metrics for AI-Assisted Development Security

| Metric | Definition | Target |
|---|---|---|
| SAST finding rate on AI-generated PRs | % of PRs with AI-generated code that produce SAST findings vs. human-only PRs | Track ratio; investigate if AI PRs have significantly higher finding rate |
| Hallucinated package blocks | Count of package installs blocked by the private registry mirror because the package did not exist in approved sources | Trending to zero (all developers aware) |
| New package review compliance | % of PRs introducing new dependencies that received a review of the new package's registry provenance | 100% |
| Secrets in AI context incidents | Count of incidents where secrets were detected in code context sent to an AI provider | Zero |
